2. Write backend/scripts/compare-baseline-reports.ts, a new committed
   comparison script (this is the "Automated comparison script, committed"
   1.7 asks for) that, per business file in docs/redesign/baseline-reports/:
   - Loads the old (pre-migration) fixture and the new post-migration one.
   - Diffs every numeric VAT field and P&L field (aggregate + each period),
     and P&L expense-category totals, flagging any delta > 0.01.
   - Cross-checks flagged deltas against docs/redesign/intentional-diffs.md
     (business 204245724's Bituach Leumi remap, ~ILS 11,775.40 P&L expense
     drop, zero VAT impact; the zero-delta duplicate-category merge) -
     anything outside that registry is a real failure.
   - Diffs the ledger: total debit/credit sums must match exactly; each
     ledger line's account code must equal its old code translated through
     ACCOUNT_CODE_MIGRATION (backend/src/bookkeeping/chart.seed.ts) -
     EXCEPT the six Bituach Leumi journal entries (ids 10000145, 10000158,
     10000167, 10000173, 10000186, 10000203), which are special-cased:
     their lines are expected on 90300, NOT on the generic old-5000 ->
     60000 mapping. Match by journalEntryId, not by old account code alone.
   - For business 204245724's P&L diff specifically: the registered
     Correction #1 delta manifests as the "הוצאות בלתי מזוהות" expense
     category vanishing entirely from the post-migration P&L (its full
     amount moved to 90300, a technical account outside P&L sections).
     Treat a category disappearing (old total present, new total absent/
     zero) as satisfying intentional-diffs Correction #1, not as a
     structural failure - but confirm the vanished amount actually equals
     the registered ~ILS 11,775.40 (amountForTax) before accepting it.
   - Prints a clear pass/fail summary per business, non-zero exit code on
     any un-registered diff.
