## Purpose
Double-entry bookkeeping core: chart of accounts, journal entries/lines, and (as of the Phase 2 categories redesign) the unified category/sub_category catalog itself — `CatalogService` is now this module's second major responsibility, not just a journal-posting helper. Posted-to by other modules (expenses, documents) — this module itself exposes no HTTP endpoints; callers (mainly `ExpensesService`) front it with their own routes.

## Key entities/files
- `account.entity.ts` — `BookingAccount` (table `booking_account`; renamed from `DefaultBookingAccount`/`default_booking_account`, Phase 1.2): the chart of accounts (code, name, type asset/liability/equity/income/expense). Per the redesign (D1 revised): carries the FULL accounting law — `vatPercent`/`taxPercent`/`reductionPercent`/`isEquipment`/`recognitionType` — plus `sectionId`/`section` (FK → `AccountingSection`), `code6111` (Form 6111 field code, NULL everywhere pending the official list), `ownerType`/`chartOwnerKey`/`accountantId`/`userId`/`businessNumber`/`visibilityScope`/`isActive` (D4 ownership model), `UNIQUE(chartOwnerKey, code)`. `pnlCategory`/`displayOrder` remain temporarily — still what `reports.service.ts` groups the P&L by — dropped in Phase 7 once reports read `sectionId` instead.
- `accounting-section.entity.ts` — `AccountingSection` (table `accounting_section`, Phase 1.1): the חתך/P&L grouping a `BookingAccount` rolls up into, replacing the `pnlCategory` string namespace going forward. Same D4 ownership columns as `BookingAccount`, `UNIQUE(chartOwnerKey, code)`.
- `category.entity.ts` / `sub-category.entity.ts` — `Category`/`SubCategory` (tables `category`/`sub_category`, Phase 2.1): the new two-table client-facing catalog, replacing `default_category`/`default_sub_category`/`user_category`/`user_sub_category` (all four now **frozen read-only**, Phase 2.5 — kept for rollback until Phase 7). `Category` is a thin display group (name, type EXPENSE/INCOME, D4 ownership). `SubCategory` is a THIN pointer at a `BookingAccount` (`accountId`, nullable) — carries no accounting law itself (D1 revised); `isPrivate` (no card, never journaled, D5) and `approvalStatus` (`APPROVED`/`MISSING_ACCOUNTING_MAPPING`/etc., D5) live here.
- `account-code-migration.entity.ts` — `AccountCodeMigration` (table `account_code_migration`, Phase 1.3): the one-time old→new account code map (`oldCode`, `newCode`, `source`) that drove Phase 1.4's renumbering. Migration artifact, not part of the core model.
- `account-code-allocator.service.ts` — `AccountCodeAllocatorService.getNextAccountCode({ownerType, type, chartOwnerKey})` (D2/1.5): allocates the next free account code in jumps of 10 within an owner's range (SYSTEM 40000-49999/60000-69999, ACCOUNTANT 50000-59999/70000-79999, CLIENT 50000-59999/80000-89999). Used by `CatalogService.findOrCreateVariantAccount` and the seeder's manual-code paths.
- `chart.seed.ts` — the SYSTEM chart as flat seed data (`ACCOUNTING_SECTIONS`, `CHART_ACCOUNTS`, `ACCOUNT_CODE_MIGRATION`): 16 sections, the renumbered chart (old 4000–6300 → 40000/60000–61300, plus 90000-range technical accounts), reviewed with Elazar (`docs/redesign/phase1-chart-review.md`). Wired into `CatalogSeedService` (Phase 2.6).
- `catalog.seed.ts` — the SYSTEM category/sub_category catalog as flat seed data (`SYSTEM_CATEGORIES`, `SYSTEM_SUB_CATEGORIES`), Phase 2.6 (D13): 12 categories / 81 sub-categories, name-keyed, transcribed from `docs/redesign/phase2-catalog-review.md` (the reviewed Phase 2.2 migration output — a portable restatement of the same data, not a second source of truth). Each sub-category row is either an `accountCode` pointer, `isPrivate: true` (D14 household bucket), or `reportScope: ANNUAL` (D14 annual-report bucket, no account by design).
- `catalog-seed.service.ts` — `CatalogSeedService` (`OnModuleInit`, replaces `AccountSeedService`/`account-seed.service.ts`/`account.seed.ts`, all deleted Phase 2.6): on every boot, (1) find-or-update `accounting_section` rows from `chart.seed.ts`, (2) find-or-update `booking_account` rows (resolving `sectionId` from `sectionCode`), (3) find-or-create SYSTEM `category` rows from `catalog.seed.ts`, (4) create-if-missing SYSTEM `sub_category` rows (never touches an existing row — an admin edit via the default-sub-category endpoints owns it from then on). All writes are explicit find-then-create/update loops, not `.upsert()` — TypeORM's `.upsert()` on a composite non-PK conflict target throws when it needs to re-fetch RETURNING columns on the UPDATE branch (found during Phase 2.6 rehearsal against `keepintax_prodcopy`). `runSeed()` is exposed publicly (separate from the `SKIP_BOOT_SEED`-gated `onModuleInit`) so standalone scripts can invoke it explicitly — see `backend/scripts/migrations/2026-07-12_run-catalog-seeder.ts`.
- `catalog.service.ts` — `CatalogService`: the single resolution + CRUD point for the D1 catalog model (Phase 2.3 read side, Phase 2.4 write side), CLIENT > ACCOUNTANT > SYSTEM precedence by name throughout (D4).
  - Reads: `getMergedCategories`/`getMergedSubCategories` (merged-by-name lists for a business context), `resolveSubCategory` (subCategoryId → full law), `resolveByName` (categoryName/subCategoryName → full law or null), `resolveAccountCode` (TRANSITION BRIDGE — old string-in/code-out signature, falls back to `'60000'`, single caller `ExpensesService.resolveAccountCode`/`buildExpenseJournalLines`), plus several scoped/single-scope lookup helpers used by `ExpensesService`'s ported CRUD.
  - Writes: `findOrCreateCategory`, `createSubCategory` (routes a legacy percent-bearing DTO through `findOrCreateVariantAccount` when no explicit `accountId`; `isPrivate`/`reportScope=ANNUAL` rows never resolve a card, always `APPROVED` with `accountId=NULL`), `updateSubCategoryLaw` (repoints `accountId` to a re-resolved/new card — never edits an existing card's percents in place, D10), `deleteCategory`/`deleteSubCategory` (soft delete, `isActive=false` — a hard delete could orphan CLIENT rows referencing a SYSTEM `categoryId`), `findOrCreateVariantAccount` (D1: "a different percentage combination = a different card" — finds an exact-law match in scope or allocates a new one via `AccountCodeAllocatorService`; **never creates a sectionless card** — inherits `sectionId` from the base account, falls back to a same-named SYSTEM section, else refuses (`null`) and the caller lands `MISSING_ACCOUNTING_MAPPING`).
- `jouranl-entry.entity.ts` — `JournalEntry` (header): globally-unique auto-increment PK plus a per-business `entryNumber` for display; date, reference (type+id back to the source expense/document), reporting period label, counter-account/sub-account codes, counterparty name, document total.
- `jouranl-line.entity.ts` — `JournalLine`: debit/credit rows under a `JournalEntry`, with VAT/tax-deductible split (`amountBeforeVat`, `vatAmount`, `taxPercent`, `vatPercent`, `amountForTax`), `isEquipment` flag.
- `dto/journal-entry-input.interface.ts` — `JournalEntryInput`/`JournalLineInput`: the shape callers (e.g. ExpensesService) pass into `BookkeepingService`.
- `bookkeeping.service.ts` — `BookkeepingService`: `createJournalEntry`, `replaceJournalEntryLines`, `updateJournalEntryFull`, `findJournalEntryNumber` — all transactional (join caller's `EntityManager` or open their own), resolve every account code before writing so a bad code aborts cleanly.
- `bookkeeping.controller.ts` — `/bookkeeping`; currently has **no routes defined** (empty controller shell — this topic is consumed entirely as an internal service today, not via its own REST surface).

## Main flows
- Journal posting is invoked programmatically by other modules (not via this module's own HTTP API): create a journal entry + lines for a new expense/document, replace lines when a source row is re-classified, or full header+lines update when a source row is edited.
- Per-business running `entryNumber` is advanced only after a successful post (via `SharedService.getJournalEntryCurrentIndex`/`incrementJournalEntryIndex`), inside the same transaction as the entry write.
- Catalog CRUD is invoked by `ExpensesService`'s category/sub-category endpoints (this module exposes no routes of its own) — `ExpensesService` maps `CatalogService`'s `Category`/`SubCategory` rows back to the legacy `categoryName`/`subCategoryName`-shaped response objects the frontend expects.
- Boot-time chart-of-accounts and catalog seeding runs automatically via `CatalogSeedService.onModuleInit` (no-op when `SKIP_BOOT_SEED=true`, e.g. for one-off scripts against `keepintax_prodcopy`).
